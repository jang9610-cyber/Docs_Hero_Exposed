# 📡 API 통신 및 보안 연동 가이드 (API Comm. & Security)

본 문서는 프로젝트의 핵심 기술인 대화형 AI(Gemini/OpenAI)의 **대화 문맥(Context) 유지**와 **API 보안 관리**를 위한 기술적 연동 계획을 정의합니다. 

> **[핵심 알림]**: 프로젝트 초기 단계에서는 **'대화 문맥 유지(Context Management)' 구현을 최우선**으로 하며, 보안용 프록시 서버 구축은 프로토타입 제작 후반부에 진행합니다.

---

## 🚩 1. 우선 순위 및 핵심 과제 (Priorities)

### **🔥 [최우선 과제] 대화 문맥(Context) 유지 및 호출**
AI API는 기본적으로 이전 답변을 기억하지 못하는 **스테이트리스(Stateless)** 방식입니다. 
-   **필요성**: 유저가 "아까는 ~라고 했잖아!"라고 추궁하거나, 이전 질문의 연장선에서 대화할 때 AI가 이를 인지해야 합니다. 문맥이 유지되지 않으면 심문 게임의 '모순 찾기' 핵심 재미가 성립되지 않습니다.
-   **구현 목표 (금일 강다영 레이서 추진)**: 유니티 내에서 `List<ChatMessage>`를 관리하여, 매 요청마다 **[시스템 프롬프트 + 누적 히스토리 + 현재 질문]**을 패키징해 전송하는 로직을 완성합니다.

### **🕒 [차순위 과제] 보안 및 프록시 서버 구축**
-   **내용**: API 키 노출 방지 및 빌드 환경에서의 호출 안정성 확보.
-   **일정**: 보안 기술 습득 및 프로젝트 안정화 이후, 프로토타입 제작 마지막 단계에 구축 예정.

---

## 🏗️ 2. 상세 구현 사양 (Implementation Specs)

### **2.1. 대화 문맥 유지 알고리즘 (Context Chain)**
AI가 전체 심문 과정을 이해하도록 하기 위해 다음과 같은 흐름을 따릅니다.

1.  **히스토리 누적**: 유저와 AI의 대화가 발생할 때마다 해당 내용을 전용 리스트(History)에 저장.
2.  **데이터 직렬화**: 서버(또는 API)로 요청을 보낼 때 단일 메시지가 아닌 **전체 리스트**를 JSON 형태로 변환.
3.  **응답 처리**: AI의 답변을 다시 히스토리에 추가하여 다음 턴의 문맥 데이터로 준비.

### **2.2. 보안 가이드 (추후 단계)**
-   **구조**: Unity (Client) ↔ Proxy Server (API Key) ↔ AI API
-   **역할**: API 키 은닉 및 요청 전달(Relay).

---

## 📝 3. 개발자 가이드 (Dev Instruction)

-   **개발자(강다영) 업무**: 금일 대화 히스토리가 정상적으로 누적되고, AI가 이전 대화 내용을 바탕으로 답변을 수행하는지 집중 테스트.
-   **데이터 구조**: `OpenAIManager` 클래스 내에 `List<ChatMessage>`를 멤버 변수로 두고, `Start` 시점에 초기화 및 `Add` 로직을 안정화할 것.

---

## ✅ 4. 기대 효과

1.  **논리적 연속성**: 유저의 질문에 따른 AI의 일관성 있는 답변 가능.
2.  **심문 게임 완성**: 이전 답변과 현재 답변 사이의 모순을 유저가 파악할 수 있는 기술적 토대 마련.
3.  **단계적 보안**: 시급한 기능(문맥)을 먼저 구현하고, 리스크가 큰 보안 요소는 기술 습득 후 안정적으로 배포.
