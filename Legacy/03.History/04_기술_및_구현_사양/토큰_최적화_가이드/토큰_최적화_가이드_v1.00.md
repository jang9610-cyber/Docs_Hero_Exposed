# [가이드] Gemini API 토큰 최적화 및 비용 효율화 전략 (v1.0)

## 1. 개요
본 문서는 Google Gemini API를 활용한 심문 시스템 구현 시, API 호출 비용을 절감 실시간 응답 속도(Latency)를 향상시키기 위한 토큰 최적화 가이드를 제공합니다. 특히 대량의 `idcarddata`가 포함되는 매 프롬프트의 효율성을 극대화하는 것을 목표로 합니다.

---

## 2. 입력 데이터(idcarddata) 구조 최적화

### **A. JSON 키값 단축 (Key Minimization)**
LLM은 JSON의 키(Key) 명칭도 토큰으로 인식합니다. 의미가 변하지 않는 선에서 최대한 단축합니다.
- **AS-IS**: `"SpeciesCharacteristics": "...", "EntryPurpose": "..."`
- **TO-BE**: `"trait": "...", "aim": "..."` (토크나이저 기준 약 30~50% 절감 가능)

### **B. 데이터 포맷팅**
형식이 엄격한 JSON보다는 불렛 포인트나 구분자(`|`)를 사용한 텍스트 형식이 토큰 효율이 더 좋을 수 있습니다.
- **예시**: `이름: 록타르 | 종족: 오크 | 목적: 수리`

### **C. 불필요한 필드 제거**
심문에 직접적인 영향을 주지 않는 정적 데이터(예: 발급 일자, 발급자 코드 등)는 `idcarddata`로 넘기지 않고 클라이언트 단에서만 관리합니다.

---

## 3. 시스템 프롬프트(System Instruction) 경량화

### **A. 역할 정의의 명확성**
중언부언하는 긴 설명은 배제하고, 핵심 페르소나와 제약 조건(Constraint)만 명시합니다.
- "당신은 마왕물산의 보안 검문을 받고 있는 몬스터입니다. 당신은 질문에 대해 반드시 사실만을 말해야 하며..." $\rightarrow$ **"너는 마왕물산 보안 검문을 받는 [몬스터]다. 사원증 데이터에 맞춰 연기하되, 의심도가 높을 때만 당황하라."**

### **B. No-Boilerplate 기법**
"이해했으면 '네'라고 답하세요" 같은 확인성 문구 등을 삭제하고, 출력 형식만 명확히 지정합니다.

---

## 4. 대화 이력(Context Window) 관리

### **A. 최근 대화 요약 (Context Swell 방지)**
대화가 길어질수록 이전 이력이 누적되어 토큰 사용량이 기하급수적으로 늘어납니다.
- **전략**: 최근 5~10회차의 대화만 유지하거나, 특정 턴마다 요약본(Summary)으로 대체하여 컨텍스트를 압축합니다.

### **B. 상태값 유지(State Management)**
NPC의 성격이나 감정 상태(Nervous, Normal 등)를 매번 긴 텍스트로 설명하지 말고, 짧은 태그(`[E:Nervous]`) 형식으로 전달합니다.

---

## 5. 생성 파라미터 제어

- **Max Output Tokens**: 심문 답변은 대체로 짧을 것이므로 150~200 토큰 정도로 제한하여 불필요한 긴 답변 생성을 방지합니다.
- **Stop Sequences**: 답변이 종료되어야 하는 지점(예: `\n`, `[End]`)을 명확히 설정하여 토큰 낭비를 막습니다.
