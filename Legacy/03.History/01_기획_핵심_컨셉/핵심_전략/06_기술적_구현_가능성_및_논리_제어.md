# [전략 제안서] 06. 기술적 구현 가능성 및 논리 제어

## 1. 개요 (Technical Feasibility)
*   **핵심 주제**: AI의 고질적인 문제인 '환각(Hallucination)'을 기술적으로 어떻게 제어하여, 유저가 신뢰할 수 있는 **'논리적 추리 퍼즐'**로서의 게임성을 확보할 것인가?

## 2. [기획 파트] 기획적 논리 및 게임성 제어 (Design Strategy)
*유저가 경험할 '추리 퍼즐'의 논리적 개연성과 재미 요소를 제어합니다.*

### **A. 에비던스 그라운딩 (Evidence-Based Grounding)**
*   **기획 의도**: AI가 지어낸 거짓말이 아닌, 제공된 '증거 문서'에 기반해 대화하도록 강제하여 추리 게임의 공정성 확보.
*   **해결 방안**: 시스템 프롬프트에 "문서에 없는 사실은 절대 지어내지 말 것"이라는 강력한 논리적 봉인 유지.


### **B. 인간적 임기응변 시스템 (Personification)**
*   **기획 의도**: 유저의 비정형적 개입(욕설, 딴소리 등)에도 AI가 성격(Trait)에 맞춰 대응하여 '살아있는 캐릭터' 느낌 전달.
*   **메커니즘**: 감정 지수(심리적 압박)와 연동하여 답변 톤을 실시간 변화시키고, 딴소리 시 다시 '심사 통과'라는 목적 대화로 유도.

### **C. 맥락 제어 및 정체성 강화 (Context Control)**
*   **기획 의도**: 대화가 길어져도 AI가 자신의 정체성(용사/몬스터)과 비밀(심사 통과 목적)을 잊지 않게(Hallucination 방지) 고정.
*   **메커니즘**: 정체성 피드백 루프를 통해 AI의 응답이 목표에서 멀어지면 시스템 프롬프트가 자동으로 궤적 수정.

## 3. [개발 파트] 기술적 구현 및 최적화 사양 (Technical Specification)
*개발자가 Unity와 API 연동 시 즉시 코드로 옮길 수 있는 기술적 사양입니다.*

### **A. 가동비 제어 및 토큰 최적화 전략 (Token Management)** 
*   **원칙**: AI 게임의 지속 가능한 서비스를 위하여 호출당 토큰 비용과 응답 속도 관리를 필수로 함.
*   **구현 전략**: 
    - **입력 전처리(Sanitization)**: 유저 입력 값의 불필요한 공백, 특수문자 등을 제거하여 잉여 토큰 소모 방지.
    - **맥락 가지치기(Context Pruning)**: 대화가 길어질 경우 핵심 논리(심문 결과)만 요약하여 유지하고, 지나간 신변잡기적 대화는 폐기하여 고정 비용 유지.

### **B. 프롬프트 엔진 및 계층형 데이터 설계 (3-Tier Layering)**
*   **Monster Identity JSON**: 개발자가 즉시 역직렬화(Deserialize)할 수 있는 표준 데이터 규약.
    ```json
    {
      "npc_id": "ORC_001",
      "base_personality": "Arrogant, Heavy Smoker",
      "secret_identity": "Hero 'Kael' looking for a Demon King",
      "suspicion_threshold": 80,
      "response_style": {
        "slang": ["Ugh...", "None of your business"],
        "panic_triggers": ["Holy Sword", "Hero Guild Receipt"]
      }
    }
    ```
*   **계층형 프롬프트 설계**:
    1.  **Layer 1 (Identity)**: NPC의 본질(이름, 성격, 배경)을 규정하는 핵심 레이어.
    2.  **Layer 2 (Knowledge: JSON)**: 서류 정보, 룰북 등 AI가 참조해야 할 '객관적 팩트'의 집합.
    3.  **Layer 3 (Dynamic Condition)**: 유저의 압박에 따른 '심리적 압박 상태(Pressure)'와 대응 기조 제어.

### **C. 유니티 연동 입출력 규약 (Unity IO Protocol)**
*   **Response Format (JSON)**: `{ "emotion": "Panic", "pressure": 85, "message": "..." }`
*   **VFX 트리거 로직**:
    - **Pressure 0~30**: 평온한 상태 (기본 연출)
    - **Pressure 31~70**: 당황 (눈동자 흔들림, 말더듬 처리)
    - **Pressure 71~100**: 극도 패닉 (**땀 흘림 이펙트 및 텍스트 떨림(Shaking)** 트리거)

## 4. 개발 리스크 및 대응 방안 (Risks & Mitigation)

| 리스크 | 대응 방안 (Mitigation) |
| :--- | :--- |
| **AI의 설정 오류 (환각)** | 데이터 세트(Data Sheet) 엄격화 및 시스템 가이드 정기 업데이트 |
| **API 응답 지연 (Latency)** | 출력 토큰 제한 및 '생각 중...' 애니메이션 등 UI 최적화 |
| **비용 관리 (Token Cost)** | **10~20% 데이터 다이어트** 및 컨텍스트 요약 기법 상시 적용 |

## 5. 결론 (Proof of Concept)
*   본 프로젝트는 단순한 '챗봇'이 아니라, **"수익성(Cost)과 게임성(Logic)을 동시에 확보한 고도화된 AI 엔진 게임"**입니다. 기술적 안정성을 위해 단계별 프롬프트 테스트와 토큰 최적화 수치를 정기적으로 모니터링하겠습니다.
