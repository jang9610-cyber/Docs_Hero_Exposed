# 데이터 구조 및 세이브 시스템 명세 (Data Structure & Save System)

> **[문서 목적]** 본 명세서는 `CharacterData`, `SessionData`, `SaveData`, `StaticData` 4종 데이터 클래스의 구조와 `DataManager`의 저장/로드 메커니즘을 코드 기반으로 정확하게 기술합니다.

## 0. 기획 의도 (Design Intent)
> 데이터 설계의 핵심 원칙은 **[런타임 데이터와 영속 데이터의 엄격한 분리]**입니다.
> `SessionData`는 '지금 이 판(一局)에서 일어나는 것', `SaveData`는 '5일에 걸쳐 누적되어야 하는 것'으로 역할을 명확히 분리하여, 세이브/로드 시 의도치 않은 데이터 누락이나 중복 누적이 발생하지 않도록 합니다.
> `DataSyncAttribute`라는 커스텀 어트리뷰트 기반 리플렉션 동기화 방식을 통해, 새로운 필드가 추가될 때마다 동기화 코드를 별도로 작성할 필요 없이 어트리뷰트 선언만으로 동기화 규칙을 정의할 수 있습니다.

---

## 1. CharacterData (NPC 개체 정보)

`CharacterData.cs`에서 정의된 NPC 1인의 모든 속성을 담는 클래스입니다. 프롬프트 조립 엔진의 핵심 입력값입니다.

| 필드명 | 타입 | 설명 |
| :--- | :--- | :--- |
| `name` | `string` | NPC 이름 (UI 표시, 프롬프트 치환) |
| `race` | `string` | 종족 한국어 표기 (UI 표시, 프롬프트 `{Race}` 치환) |
| `raceType` | `RaceTypes` (enum) | 종족 데이터 매칭용 열거형 |
| `position` | `string` | 직책/지위 (프롬프트 `{Position}` 치환) |
| `purpose` | `string` | 출입 목적 (프롬프트 `{Purpose}` 치환) |
| `idCardImage` | `string` | 신분증 초상화 스프라이트 경로 |
| `standingImage` | `string` | 씬 내 스탠딩 이미지 스프라이트 경로 |
| `intelligence` | `IntelligenceLevel` (enum) | 지능 레벨 (프롬프트 능력치 레이어에서 행동 지침 결정) |
| `maxStress` | `int` | (레거시 필드, 현재 `MAX_STRESS=200` 상수로 대체) |
| `npcType` | `IdentityTypes` (enum) | NPC 유형 — 판정 로직과 보상 계산의 핵심 |
| `npcTypeId` | `string` | NPC 유형 문자열 ID |
| `isPass` | `bool` | 통과 가능 여부 (true=진짜 직원, false=위장 용사) |
| `knowledgeType` | `KnowledgeType` (enum) | **프롬프트 분기의 핵심** |

### 1.1. KnowledgeType 4종 분기 (프롬프트 엔진 분기 기준)

| 값 | 한국어 명칭 | 프롬프트 동작 |
| :--- | :--- | :--- |
| `PerfectTruth` | 완전 파악 | 기본 정보 + 실제 일일 메모 제공 |
| `MissingBasicInfo` | 신상 망각 | 기본 정보 없음(창작 모드) + 실제 일일 메모 제공 |
| `FakeDailyInfo` | 정보 오염 | 기본 정보 정상 + 가짜 일일 메모 제공 |
| `CompleteFake` | 완전 위장 | 기본 정보 없음(창작 모드) + 가짜 일일 메모 제공 |

### 1.2. IdentityTypes (NPC 유형) — 보상 분기 기준

| 값 | 명칭 | PASS 선택 시 | EJECT 선택 시 |
| :--- | :--- | :--- | :--- |
| `T_DEMON` | 마족 직원(진짜) | +100G, +1 신뢰 | 0G, -1 신뢰 |
| `T_HUMAN` | 인간 침입자(위장) | 0G, -1 신뢰 | +100G, +1 신뢰 |
| `T_INSPECTOR` | 감사관 (함정) | +100G, -5 신뢰 | +100G, +5 신뢰 |
| `T_SMUGGLER` | 밀수꾼 (뇌물 제안) | +500G, -1 신뢰 | +100G, +1 신뢰 |

> **SMUGGLER 뇌물 메커니즘**: PASS 선택 시 `slushFund += 500` 누적. 5일 차 종료 시 `TotalBribe >= brideThreshold(2000)`를 충족하면 엔딩 분기에 영향.

## 2. SessionData (세션 내 런타임 데이터)

한 세션(하루치 플레이) 동안만 유지되는 임시 데이터입니다. 씬 전환에도 유지되지만 `DataManager.ClearSessionData()` 호출 시 초기화됩니다.

| 필드명 | 초기값 | DataSync 동기화 방식 | 설명 |
| :--- | :--- | :--- | :--- |
| `DayState` | 1 | 덮어쓰기 저장, 로드 시 복원 | 현재 일차 |
| `npcList` | `List<CharacterData>()` | 비동기화 | 금일 NPC 순서 목록 |
| `maxNpcNum` | 5 | 비동기화 | 일일 최대 NPC 수 |
| `nowNpcIdx` | 0 | 비동기화 | 현재 처리 중인 NPC 인덱스 |
| `nowNpcInfo` | null | 비동기화 | 현재 NPC `CharacterData` 참조 |
| `chatHistory` | `List<Message>()` | 비동기화 | NPC 대화 기록 (UI 표시용) |
| `playerMoney` | 0 | **누적 저장**, 로드 시 복원 | 보유 자금 |
| `trustPoint` | 0 | **누적 저장**, 로드 시 복원 | 신뢰 포인트 |
| `slushFund` | 0 | **누적 저장**, 로드 시 비복원 | 당일 비자금 수입 |
| `correctCounts` | 0 | **누적 저장**, 로드 시 비복원 | 당일 정답 수 |
| `wrongDecision` | 0 | **누적 저장**, 로드 시 비복원 | 당일 오판 수 |
| `passInspector` | 0 | **누적 저장**, 로드 시 비복원 | 당일 감사관 통과 수 |
| `playerItems` | `ItemData[3]` | 별도 직렬화 | 인벤토리 (최대 3슬롯) |

> **DataSyncAttribute 동작**: `accumulateOnSave: true`는 저장 시 기존 SaveData 값에 더함(5일간 누적). `loadToSession: true`는 로드 시 SaveData → SessionData로 불러옴.

## 3. SaveData (영속 저장 데이터)

`JsonUtility.ToJson(saveData)`로 직렬화하여 `Application.persistentDataPath/saveFileName`에 저장됩니다.

| 필드명 | 초기값 | 설명 |
| :--- | :--- | :--- |
| `DayState` | 1 | 현재 일차 (로드 시 SessionData에 복원) |
| `TotalMoney` | 0 | **5일간 누적 자금** (SessionData.playerMoney 누적) |
| `trustPoint` | 0 | **5일간 누적 신뢰 포인트** |
| `TotalBribe` | 0 | **5일간 누적 비자금** (엔딩 분기 조건) |
| `TotalCorrectCounts` | 0 | 5일간 총 정답 수 |
| `TotalWrongDecision` | 0 | 5일간 총 오판 수 |
| `playerItemNames` | `List<string>()` | 아이템 이름 목록 (ScriptableObject 직렬화 불가로 이름만 저장) |
| `playerSettings` | `PlayerSettings()` | 볼륨, 전체화면 등 사용자 환경 설정 |

> **아이템 역직렬화 특이사항**: `ItemData` ScriptableObject는 JSON으로 직접 저장할 수 없으므로 이름(`string`)으로만 저장 → 로드 시 `staticData.GetItemDataByName(name)`으로 재매핑.

## 4. StaticData (기획 수치 정적 데이터)

매 게임 시작 시 `Resources.LoadAll()`로 로드되는 불변 기획 수치입니다.

| 필드명 | 실제 초기값 | 설명 |
| :--- | :--- | :--- |
| `normalModeDate` | **5** | 게임 총 일차 수 |
| `addMoney` | **100** | 일반 판정 성공 보수 |
| `addMoney_Smuggler` | **500** | 밀수꾼 PASS 시 뇌물 획득량 |
| `addMoney_Inspector` | **100** | 감사관 처리 시 보수 (PASS/EJECT 동일) |
| `dayTax` | **-200** | 일일 고정 세금 차감량 |
| `addTrustPoint` | **1** | 일반 판정 성공/실패 시 신뢰 증감량 |
| `addTrustPoint_Smuggler` | **1** | 밀수꾼 판정 시 신뢰 증감량 |
| `addTrustPoint_Inspector` | **5** | 감사관 판정 시 신뢰 증감량 |
| `brideThreshold` | **2000** | 비자금 엔딩 조건 (TotalBribe >= 2000) |
| `ejectsThreshold` | **15** | 신뢰 엔딩 조건 (trustPoint >= 15) |

> **경제 명세서 보정**: 기존 경제 명세서에 기재된 '감사관 벌금 -400G'는 기획 의도 수치이고, 실제 코드(`StaticData`)의 `addMoney_Inspector = 100`으로 양방향 동일 보수 지급이 구현되어 있음.

## 5. 저장/로드 흐름

```
[게임 시작]
    DataManager.Awake()
        → SessionData = new SessionData()   // 세션 초기화
        → SaveData = new SaveData()         // 세이브 초기화
        → staticData.LoadItemData()         // Resources 로드
        → staticData.LoadMemoData()
        → staticData.LoadMenualData()

[이어하기]
    DataManager.LoadData()
        → File.ReadAllText(saveFilePath) → JsonUtility.FromJson<SaveData>()
        → SyncSessionWithSave()             // SaveData → SessionData 동기화
            └─ AutoSyncData(리플렉션)
            └─ 아이템 이름 → ItemData 역직렬화

[정산 씬 퇴근]
    SettlementController.LeaveWork()
        → sessionData.DayState++
        → DataManager.SaveData()
            → SyncSaveWithSession()         // SessionData → SaveData 동기화
                └─ AutoSyncData(리플렉션, 누적 처리)
                └─ 아이템 ItemData → 이름 직렬화
            → JsonUtility.ToJson(saveData) → File.WriteAllText()

[엔딩 진입]
    EndUIManager.Awake()
        → RequestRemoveSavedata()           // 세이브 파일 삭제 (다음 시작 시 새 게임)
```
